## 2. C Synthesis 当前状态

### 2.1 当前结果

`streaming_frontend` 的 C Synthesis 尚未成功，因此目前不能记录或使用其资源、延迟、II 和时序结果。

报错：

```text
ERROR: [SYN 201-102] Cannot find function 'streaming_frontend' in design source files.
```

### 2.2 根本原因

当前工程目录仍使用旧项目名称和旧 Solution：

```text
07_experiments/stage2_hls_streaming_accelerator/conv3x3_core/conv3x3_core/solution1/
```

检查发现配置不一致：

- `solution1/script.tcl` 设置了：

  ```tcl
  set_top streaming_frontend
  ```

- 但 `solution1/directives.tcl` 仍然保留：

  ```tcl
  set_directive_top -name conv3x3_core "conv3x3_core"
  ```

因此 Vitis HLS 综合过程中先处理旧的 `conv3x3_core`，之后又尝试寻找 `streaming_frontend`，造成 Top Function 配置冲突。

这不是 `generic_floor<double>` 信息导致的错误；那只是旧综合过程中的普通信息。真正错误是 Top Function 与自动生成的 directive 不一致。

### 2.3 当前已有的旧综合报告

已有报告属于旧 Top Function `conv3x3_core`，不能作为 `streaming_frontend` 的最终报告：

- 估计时钟周期：6.442 ns，未达到 5 ns 目标。
- 顶层延迟：1,747,156,673 cycles。
- 内部循环 II：17，不是顶层整体 II，也不能作为 II=1 证据。
- 资源：BRAM 2、DSP 13、FF 9303、LUT 9221。

这些数据只能作为旧版本参考，不能直接写入最终论文的 streaming frontend 结果。

## 3. 当日遇到的问题与处理

### 问题一：C Simulation 初始误差较大

曾出现较大的 MSE，随后定位到实现与 golden 数据的定点量化/饱和处理不一致。通过调整卷积核心的定点计算和饱和逻辑，最终实现：

```text
MSE: 0
Max absolute error: 0
```

### 问题二：多个 Test Bench 同时参与编译

曾出现多个 `main` 函数重复定义。原因是 `tb_conv3x3_core.cpp`、`tb_line_buffer.cpp`、`tb_window_buffer.cpp`、`tb_fsm_controller.cpp` 和 `tb_streaming_frontend.cpp` 被同时加入同一个 C Simulation。

处理原则：每次 C Simulation 只保留当前目标模块对应的一个 Test Bench。源文件本身不需要删除。

### 问题三：`streaming_frontend` C Synthesis 找不到顶层函数

已确认是旧 Vitis HLS 工程的 `directives.tcl` 残留了 `conv3x3_core` Top Function 配置。不要删除 `.h` 或 `.cpp` 源文件；明天应新建干净的 HLS 工程或重新创建 Solution。

## 4. 必会知识点

1. C Simulation 只验证 C/C++ 功能和数值正确性，不产生可靠的硬件资源与时序结果。
2. C Synthesis 才能生成 LUT、FF、DSP、BRAM、Latency、II 和时序估计报告。
3. Test Bench 中的 `main()` 不属于综合设计源，不能把多个 Test Bench 一起参与同一次 C Simulation。
4. HLS 工程的 Top Function 必须同时与 GUI、`script.tcl` 和 directive 配置一致。
5. `Line Buffer` 用于缓存卷积窗口所需的历史行；`Window Buffer` 用于维护当前 3×3 滑动窗口；FSM 控制填充和处理时序。
6. MSE = 0 只能说明当前测试数据上的功能输出完全匹配，不能说明已经达到 II=1 或 200 MHz。
