## 📌 阶段一：YOLO Baseline 建立与信息熵裁剪（7月4日 - 7月21日）【18天】

### 📌 目的

建立 YOLOv8n 在 NEU-DET 上的性能基线，**用信息熵量化** FPN 各跳连分支的贡献，裁剪低信息分支，实现精度无损的 FPN 结构轻量化，同时为 HLS 阶段导出黄金数据。

### 🧪 实验操作

| 实验编号 | 操作 | 说明 |
|---------|------|------|
| **A1 (Baseline)** | YOLOv8n 在 NEU-DET 上训练 100 epochs | 记录 mAP@0.5, FLOPs, Params |
| **A2 (Entropy)** | 信息熵引导裁剪 + 50 epochs Fine-tune | **本文方法** |
| **A3 (Random)** | 随机裁剪（3 次取均值） | 对照 |
| **A4 (L1-Norm)** | L1 范数权重裁剪 | 对照 |
| **B1 (数据导出)** | 导出 golden_input.txt / golden_output.txt / weights.txt | HLS 验证用 |

### 📋 实验步骤

**Subphase 1.1 — 数据集准备（7/4 - 7/6）**
1. 下载 NEU-DET（官方链接），6 类缺陷：crazing, inclusion, patches, pitted_surface, rolled-in_scale, scratches
2. 按 7:2:1 划分 train/val/test
3. Python 脚本标准化切片（640×640）+ 在线增强（Mosaic + 随机翻转 + HSV 抖动）

**Subphase 1.2 — Baseline 训练（7/7 - 7/11）**
4. `git clone ultralytics/ultralytics`，使用 `yolov8n.pt` 预训练权重
5. 修改 `data.yaml` 指向 NEU-DET
6. 训练 100 epochs，`batch-size=16`，`imgsz=640`
7. 记录 **Baseline-1**：mAP@0.5, FLOPs, Params, CPU/GPU Latency

**Subphase 1.3 — FPN 结构分析（7/12 - 7/14）**
8. 定位 YOLOv8 Neck 的 FPN/PAN 融合层，识别所有跳连分支（Skip Connection）
9. 编写 Hook 脚本提取每路跳连的前向传播特征图 Tensor
10. 对每个 Tensor 逐通道计算信息熵：

    **Step 1 — 特征图提取：** 设第 $k$ 个跳连分支输出的特征图为 $X^{(k)} \in \mathbb{R}^{C_k \times H_k \times W_k}$
    
    **Step 2 — 逐通道空间熵计算：** 对每个通道 $c \in \{1, \dots, C_k\}$，将特征图值归一化至 $[0, 1]$ 区间后以 256-bin 直方图估计概率质量函数 $p_c(b)$（$b=0,\dots,255$），计算该通道的空间信息熵：
    $$H_c^{(k)} = -\sum_{b=0}^{255} p_c(b) \cdot \log_2\big(p_c(b) + \epsilon\big), \quad \epsilon = 10^{-8}$$

    **Step 3 — 分支级聚合：** 该跳连分支的总信息量定义为所有通道熵的均值：
    $$\bar{H}^{(k)} = \frac{1}{C_k}\sum_{c=1}^{C_k} H_c^{(k)}$$

    **Step 4 — 裁剪阈值：** 对所有跳连分支的 $\bar{H}^{(k)}$ 排序，裁剪掉信息量处于**下四分位数（Q1）以下**的分支。若分支数少于 5 个，则仅裁剪 $\bar{H}^{(k)}$ 最低的 1 个分支

**Subphase 1.4 — 三组裁剪实验（7/15 - 7/19）**
11. 按信息熵排序 → 裁剪 Q1 以下分支 → Fine-tune 50 epochs → 记录 **A2**
12. 随机裁剪（相同分支数）× 3 次取平均 mAP（排除随机性干扰）→ 记录 **A3**
13. L1 权重范数裁剪（对每个跳连分支的卷积核权重计算 $\|W^{(k)}\|_1 = \sum\|w_i\|$，裁剪 L1 范数最低的相同数量分支）→ 记录 **A4**
14. **Table 2 数据固化：**

| Pruning Strategy | Branches Pruned | mAP@0.5 | ΔmAP | FLOPs | Params |
|-----------------|----------------|---------|------|-------|--------|
| Random (avg×3)  | N              | XX.X    | -X.X | X.XM  | X.XM   |
| L1-Norm         | N              | XX.X    | -X.X | X.XM  | X.XM   |
| Entropy (Ours)  | N              | XX.X    | -X.X | X.XM  | X.XM   |

**Subphase 1.5 — 黄金数据导出（7/20 - 7/21）**
15. 注册 Hook 提取裁剪后网络的融合层输入 Tensor 和输出 Tensor
16. 保存为 `golden_input.txt`、`golden_output.txt`、`weights.txt`（均为纯文本，16 位定点格式）

### 🔬 预期结论

> **"基于信息熵的 FPN 分支裁剪在保持 mAP 损失 < 1.5% 的前提下，剪除了 FPN 中信息冗余的跳连分支，减少了约 XX% 的 Feature Map 跨层搬运量。与随机裁剪（ΔmAP = -X.X%）和 L1 裁剪（ΔmAP = -X.X%）相比，信息熵裁剪的精度损失最小，验证了信息熵作为裁剪指标的有效性。"**

## 📌 阶段二：HLS 核心算子流式重构（7月22日 - 8月14日）【24天】🔥 核心攻坚期

### 📌 目的

将裁剪后的 FPN 融合算子（Concat + 3×3 Conv）用 HLS C++ 实现为 AXI4-Stream 流式硬件，采用 Line Buffer + Window Buffer + FSM + PIPELINE 架构，使启动间隔 II → 1，并将 C++ 输出与 Python 黄金数据精确对齐。

### 🧪 实验操作

| 实验编号 | 操作 | 说明 |
|---------|------|------|
| **B2** | 数据类型定点化（float → ap_fixed<16,8>） | 精度验证（MSE ≤ 1e-6） |
| **B3** | Line Buffer 独立验证 | 行滚动逻辑正确性 |
| **B4** | Window Buffer + FSM 联调 | 窗口滑动 vs Python 参考一致 |
| **B5** | PIPELINE + ARRAY_PARTITION 全开 | II=1 验证 |

### 📋 实验步骤

**Subphase 2.1 — 数据类型与接口（7/22 - 7/25）**
1. 用 C++ 重写 Concat + 3×3 Conv 融合算子
2. 所有 float 改为 `ap_fixed<16,8>`（先固定 16 位，阶段三再做位宽扫描）
3. 添加 `#pragma HLS INTERFACE axis port=in/out` 配置 AXI4-Stream
4. C Simulation 验证输出与 `golden_output.txt` 的 MSE ≤ 1e-6

**Subphase 2.2 — Line Buffer 独立验证（7/26 - 7/28）**
5. 用 C++ 二维数组实现 K-1=2 行的环形队列（Line Buffer），缓存 K-1 行像素
6. 先不接入 FSM 和 Window Buffer——仅用最简单的 `memcpy` 式读写验证 Line Buffer 的行滚动（Row-Shifting）逻辑
7. C++ Testbench 打印缓冲区内容与 Python 滑动窗口的逐行缓存比对

**Subphase 2.3 — Window Buffer + FSM（7/29 - 8/2）**
8. 在 Line Buffer 之上实现 3×3 全并行寄存器阵列（Window Buffer）：每入一个新像素，窗口内 9 个寄存器同时向相邻位置移位，最旧像素被移出
9. 设计三阶段有限状态机（`IDLE → FILL_BUFFER → PROCESS`），状态转移条件用枚举常量定义，每个状态的输出信号和次态逻辑写在独立 `switch-case` 中
10. 在极简测试用例（如 5×5 全 1 矩阵做 3×3 卷积）上调通 FSM
11. C Simulation 验证窗口内像素与 Python 端滑动窗口取值完全一致

**Subphase 2.4 — 三模块联调 + 黄金数据对齐（8/3 - 8/6）**
12. Line Buffer + Window Buffer + FSM 串联，编写 C++ Testbench
13. 读取阶段一导出的 `golden_input.txt`，逐像素送入流水线
14. 将输出矩阵与 `golden_output.txt` 逐元素比对，MSE ≤ 1e-6（定点化舍入误差范围内）

**Subphase 2.5 — 并行度极致压榨（8/7 - 8/14）**
15. 注入 `#pragma HLS PIPELINE II=1` 展开硬件流水线
16. 解决内存端口并发冲突 → `#pragma HLS ARRAY_PARTITION variable=line_buf complete dim=1` 强行切分 BRAM
17. C Synthesis → 记录资源报告：LUT / FF / DSP / BRAM / Latency / II
18. 尽量跑通 C/RTL Cosimulation，波形图验证

### 🛡️ 硬件调试缓冲带

若 8月12日前仍未通过 Cosimulation，按以下优先级降级：

> **"所提出的流式卷积核通过 Line Buffer 将 DDR 访问次数从 O(N) 降至 O(1)，Window Buffer 实现 3×3 窗口的全并行寄存器移位，FSM 三状态机精确控制流水线节拍。在 200 MHz 目标频率下，C Synthesis 报告显示启动间隔 II=1，总延迟为 XXX 个时钟周期，资源消耗为 LUT: XXK, DSP: XX, BRAM: XX KB，满足 Zynq-7020 的边缘部署约束。"**

## 💡 导师冷峻红线提示：执行过程中的三大"不可触碰边界"

1. **绝对禁做全栈网络：** 在 HLS 阶段，**只将 YOLO FPN 中最吃带宽的那一层大尺度融合算子（卷积层）抽出来写 C++**。整个网络的主干（Backbone）依然留在 Python 里跑。叙事是"针对多尺度关键瓶颈算子进行流水线重构"，试图用 C++ 写完整个 YOLO 几十层网络，3 个月敏捷周期会在工程地狱里崩溃。

2. **严防"伪专注"与自我感动：** 进入阶段二后，每天 6 小时必须是坐在电脑前**开着 Vivado HLS 和 PyTorch 调代码、看报告的硬核时间**。遇到 `II Violation` 警告不要逃避，去查内存端口冲突。任务没有产出数据，就代表这一天的工作量为零。

3. **保持图表基准的绝对公平：** 做 C 组和 D 组的对比时，所有模型必须跑在**完全相同的 NEU-DET 数据集和相同的物理分辨率**下。审稿人如果发现你对比别人论文时偷偷换了更容易跑的数据集，会直接判定学术不端予以秒拒（Desk Reject）。
