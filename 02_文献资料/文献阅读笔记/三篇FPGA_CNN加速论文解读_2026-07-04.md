## 1. Toupas et al., 2024 - SMOF: Streaming Modern CNNs on FPGAs with Smart Off-Chip Eviction

### 1.1 论文解决的问题

这篇论文解决的是：现代 CNN 拓扑在 FPGA 流式架构中很难完全放进片上存储的问题。

传统 FPGA CNN 流式架构通常希望把层间数据都留在片上，通过 FIFO、BRAM、URAM 让不同层形成流水线。这样可以减少 DDR 访问，提高吞吐率。但是现代网络不再是简单的顺序结构，UNet、YOLO、X3D 这类网络都有长跳连、分支、Concat、层级式融合。长跳连意味着早期特征图要被保存很久，直到后面某个层再参与融合。若全部用片上 FIFO 保存，片上存储会很快成为瓶颈。

论文的核心问题可以概括为：

```text
现代 CNN 长跳连 / 分支结构
        ↓
需要保存大量中间 activation 和权重
        ↓
片上 BRAM / URAM / FIFO 不够
        ↓
传统全片上 streaming 架构无法映射大模型
        ↓
需要在吞吐、延迟、片上资源和片外带宽之间重新分配数据
```

这不是一个“算得不够快”的问题，而是“数据留在哪里、什么时候搬运、搬运多少才不会拖垮流水线”的问题。

### 1.2 核心解决方法

#### 1.2.1 总体路线图

```text
输入：CNN 计算图 G、目标 FPGA 资源 R、片外带宽 BW、batch size b
        ↓
把 CNN 抽象成 DAG：节点是 Conv/Pool/Activation/Concat，边是 feature map 数据流
        ↓
识别造成片上缓存压力的长跳连、深 FIFO、权重存储
        ↓
方法 1：Activation Eviction，把部分 activation 暂存到片外 DDR
        ↓
方法 2：Weight Fragmentation，把部分权重变成运行时片外加载
        ↓
方法 3：Graph Partitioning，把网络拆成多个子图并利用 FPGA 重配置
        ↓
方法 4：DSE，在资源约束和带宽约束下搜索最优设计
        ↓
输出：满足片上资源、片外带宽、依赖顺序的 streaming accelerator
```

#### 1.2.2 算法一：Activation Eviction

**解决什么问题：** 长跳连需要在片上保存大量 activation，导致 FIFO/BRAM 深度过大。

传统做法：

```text
早期层输出 activation
        ↓
长时间保存在片上 FIFO/BRAM
        ↓
后面 Concat / Add / Merge 再读取
```

SMOF 的做法：

```text
早期层输出 activation
        ↓
只在片上保留小 FIFO，用于 DMA burst 对齐
        ↓
大部分 activation 写到片外 DDR
        ↓
到后续 merge point 时再从 DDR 读回
```

算法思想是把“深片上 FIFO”替换成“浅片上 FIFO + 片外环形缓存”。对长跳连而言，activation 本来就要等待很多层之后才用，与其占用昂贵片上存储，不如用 DDR 承担容量压力。但这个方法不能随便用，因为 DDR 访问会增加带宽压力。论文因此用节省的片上存储量和新增片外带宽开销来评价是否值得 eviction。

简化判断逻辑：

```text
如果某条边保存 activation 会消耗大量片上深度 d，
并且把它搬到片外后新增带宽 ΔBW 仍小于平台带宽上限，
则该边适合 activation eviction。
```

论文还支持 RLE、Huffman 等无损编码来减少片外带宽。它的重点不是压缩模型精度，而是压缩或调度数据传输。

#### 1.2.3 算法二：Weight Fragmentation

**解决什么问题：** streaming 架构通常希望每层权重都常驻片上，但大模型参数多，片上存储不足。

传统做法：

```text
每个 Conv/MatMul 节点拥有专用片上权重存储
所有权重常驻片上
```

SMOF 的做法：

```text
把权重存储分成 static region 和 dynamic region
        ↓
static region：常驻片上
        ↓
dynamic region：运行时从片外 DDR 加载
        ↓
不同层或不同子图共享一段物理片上存储
```

其中 dynamic ratio `m` 是设计变量：

```text
m = 0：所有权重都在片上，不需要片外加载
m = 1：所有权重都从片外加载
0 < m < 1：一部分片上，一部分片外
```

直观理解：Weight Fragmentation 用片外带宽换片上容量。若某层权重占用大但访问节奏可预测，就可以把一部分权重转移到 DDR，在流水线运行时按需加载。

#### 1.2.4 算法三：Graph Partitioning / Reconfiguration

**解决什么问题：** 当整个网络无法一次性放入 FPGA 时，可以把网络拆成多个子图。

方法：

```text
完整 CNN 图 G
        ↓
切分为 S1, S2, ..., SN
        ↓
每个子图生成一个定制硬件设计
        ↓
按拓扑依赖顺序执行
        ↓
必要时进行 FPGA 重配置
```

其代价是重配置时间。batch 越小，重配置时间在总延迟中的比例越高；batch 越大，重配置开销被摊薄，但单帧延迟可能变大。因此它提供的是 latency-throughput trade-off。

#### 1.2.5 算法四：Design Space Exploration, DSE

论文的 DSE 算法把每个节点的设计变量写成：

```text
Dv = {
  operation parallelism p,
  activation input eviction flag ai,
  activation output eviction flag ao,
  weight fragmentation ratio r
}
```

搜索流程：

```text
1. Resource-minimal initialization
   从最省资源的配置开始，尽可能多地拆分子图，每个节点使用最低并行度。

2. Compute parallelism allocation
   找到当前流水线中最慢的操作，优先增加它的并行度，以降低 pipeline II。

3. On-chip memory allocation
   在 BRAM、LUTRAM、URAM 之间分配 activation 和 weight 存储，使资源利用更均衡。

4. Off-chip bandwidth allocation
   按照“节省片上存储 / 增加片外带宽”的收益比排序，优先把最划算的 activation 或 weight 搬到片外。

5. Partition merging
   尝试合并子图，减少重配置次数；若合并后资源或带宽超限，则不合并。
```

这个 DSE 的关键点是：不是单独优化计算，也不是单独减少 DDR 访问，而是在片上资源、片外带宽、并行度、重配置开销之间做联合搜索。

### 1.3 论文结论与不足

#### 结论

1. 对具有复杂跳连的现代 CNN，传统全片上 streaming 架构会受到片上存储限制。
2. Activation Eviction 和 Weight Fragmentation 可以把部分片上存储压力转移到片外 DDR，同时尽量不破坏流水线。
3. DSE 可以在给定 FPGA 资源和带宽限制下自动寻找更合理的映射方案。
4. 实验覆盖 UNet、YOLOv8、X3D、UNet3D 等模型，并报告了最高可达 10.65x 的吞吐提升；在 YOLOv8 任务上也展示了明显 fps 改善。

#### 不足

1. 该方法依赖片外带宽。如果目标板卡 DDR 带宽较弱，eviction 可能把片上存储瓶颈转移成片外带宽瓶颈。
2. 它的核心是系统级映射和工具流优化，不直接回答“哪些 FPN 分支语义上冗余”。
3. Activation eviction 是保留原网络结构再优化存储，而不是从算法结构上减少低价值特征流。
4. DSE 复杂度较高，更适合自动化工具流；对个人论文项目而言，完整复现成本较大。
5. 对低端 Zynq-7020 这类资源更紧张的平台，SMOF 中大规模 streaming toolflow 的部分结果不能直接迁移，需要降级为局部算子级借鉴。

### 1.4 与我的论文的联系

这篇论文与你的论文联系非常强。

你的论文问题可以借用 SMOF 的痛点表述：

```text
YOLO/FPN 的多尺度融合和跳连结构会造成大量 feature map 搬运。
这些 feature map 如果长期保存在片上，会占用 BRAM；如果频繁写回 DDR，会造成带宽压力。
```

但你的解决思路与 SMOF 不同：

```text
SMOF：保留复杂拓扑，通过 smart off-chip eviction 重新安排 activation/weight 存储。
你的论文：先用信息熵识别低价值 FPN 分支，从算法结构上减少低收益特征流；再对保留下来的核心融合算子做 HLS 流式重构。
```

可以放入论文的位置：

1. **Chapter 1 Introduction**：引用它说明现代 YOLO/FPN/UNet 这类长跳连网络会带来片上存储压力。
2. **Chapter 2 Related Work - FPGA-Based CNN Accelerators**：作为现代 streaming FPGA toolflow 的代表。
3. **Chapter 2 Research Gap**：指出现有方法多从存储调度角度处理跳连，缺少面向 FPN 分支信息量的算法级裁剪。
4. **Chapter 4 SOTA Comparison**：SMOF 的 YOLOv8 FPGA 结果可作为 SOTA 表格候选，但要注意平台、精度、数据集和模型不同。

## 2. Yang et al., 2023 - A Lightweight Detection Method for Remote Sensing Images and Its Energy-Efficient Accelerator on Edge Devices

### 2.1 论文解决的问题

这篇论文解决的是：遥感目标检测网络精度高但结构复杂，难以部署到低功耗、资源受限的边缘 FPGA 设备上。

遥感图像目标检测有几个困难：

```text
图像分辨率高
        ↓
目标尺度变化大，小目标多
        ↓
检测网络需要较强特征提取能力
        ↓
YOLO 等网络参数量、计算量、特征图访存量大
        ↓
低功耗边缘设备难以实时运行
```

论文认为，只做硬件加速不够，因为原始网络太重；只做轻量化也不够，因为推理仍需要专门硬件支持。因此它提出软硬件协同：

```text
算法端减小网络
        +
硬件端设计高能效 CNN accelerator
```

### 2.2 核心解决方法

#### 2.2.1 总体路线图

```text
原始 YOLOv3
        ↓
用 GhostNet 替换 Darknet-53 backbone
        ↓
得到 Ghost-YOLO
        ↓
基于 BN scaling factor 做通道剪枝
        ↓
得到 Ghost-YOLOP
        ↓
用未剪枝模型作为 teacher，对剪枝模型做知识蒸馏恢复精度
        ↓
进行 16-bit 动态定点量化
        ↓
面向 Ghost-YOLOS 结构设计 FPGA accelerator
        ↓
在 Xilinx ZYNQ Z7020 上部署并评估功耗、吞吐、DSP 效率
```

#### 2.2.2 算法一：Ghost Module / GhostNet 轻量化

**解决什么问题：** 普通卷积会生成大量通道，其中很多特征图存在冗余。

普通卷积：

```text
输入特征图 X
        ↓
用大量卷积核直接生成 M 个输出通道
        ↓
计算量高，参数量高
```

Ghost Module：

```text
先用少量普通卷积生成一部分 intrinsic feature maps
        ↓
再用廉价线性操作生成更多 ghost feature maps
        ↓
拼接得到完整输出
```

直观理解：不是所有输出通道都需要昂贵卷积直接生成，部分通道可以由已有特征经过轻量变换得到。论文用 GhostNet 替换 YOLOv3 的 Darknet-53 backbone，从源头降低计算量和参数量。

#### 2.2.3 算法二：基于 BN 的通道剪枝

**解决什么问题：** Ghost-YOLO 虽然比 YOLOv3 小，但对低成本 FPGA 仍然偏重。

BN 层中有 scaling factor，通常记为 gamma。若某个通道的 gamma 接近 0，说明该通道在归一化后对输出影响很小。论文利用这一点裁剪通道：

```text
训练 Ghost-YOLO
        ↓
读取 BN 层每个通道的 scaling factor gamma
        ↓
按 |gamma| 排序
        ↓
删除 gamma 接近 0 的通道及其连接
        ↓
得到更窄的网络 Ghost-YOLOP
```

这个方法的优点是简单，容易实现，能直接减少通道数、参数量和计算量。缺点是它关注通道重要性，不直接关注 FPN/Neck 中跨层分支的信息贡献。

#### 2.2.4 算法三：知识蒸馏恢复精度

**解决什么问题：** 通道剪枝会损失精度。

论文用 teacher-student 框架：

```text
Teacher：未剪枝的 Ghost-YOLO
Student：剪枝后的 Ghost-YOLOP
        ↓
Student 不只学习真实标签
还学习 Teacher 的输出分布、中间层表示和边框回归结果
```

它的损失包含三类：

```text
classification loss：类别预测蒸馏
regression loss：边界框回归蒸馏
intermediate-layer loss：中间层特征蒸馏
```

直观理解：剪枝后的小模型容量下降，蒸馏相当于让大模型把“暗知识”传给小模型，使小模型在参数更少的情况下尽量保持检测性能。

#### 2.2.5 算法四：16-bit 动态定点量化

**解决什么问题：** FPGA 上浮点计算资源代价高，定点更适合低功耗部署。

论文将权重、bias、输入输出特征图和中间结果转为 16-bit 动态定点。其核心是为每组数据寻找合适的 exponent-marker，使量化误差最小：

```text
遍历候选 exponent-marker
        ↓
把浮点值 Q_float 量化为 Q_(bw, expw)
        ↓
计算量化误差
        ↓
选择误差最小的 exponent-marker
```

该方法的思想是：总位宽固定为 16-bit，但小数点位置可根据数据动态调整，从而兼顾范围和精度。

#### 2.2.6 算法五：面向 Ghost-YOLOS 的 FPGA Accelerator

硬件端包括几个关键设计：

```text
DDR：存储 feature maps 和 weights
        ↓
BRAM buffer：缓存当前计算块
        ↓
Ping-pong / dual cache：读下一块数据的同时计算当前块
        ↓
Reconfigurable PE array：支持普通卷积、depthwise convolution 等不同模式
        ↓
Adder tree：累加 PE 输出
        ↓
BN + Leaky ReLU / SE / pooling 后处理
        ↓
输出结果写回
```

其中最值得注意的是：

1. **双缓存机制**：用两组 buffer 交替读写，隐藏 DDR 传输延迟。
2. **分块存储**：按高、宽、通道对 feature map 分块，让 BRAM 访问更符合卷积窗口读取顺序。
3. **可重构 PE 阵列**：同一组计算资源根据控制信号执行普通卷积或 depthwise 卷积，避免为每种算子都复制一套硬件。
4. **SE 模块硬件友好化**：对 Sigmoid 等不适合硬件的函数做近似实现，降低资源消耗。

### 2.3 论文结论与不足

#### 结论

1. Ghost-YOLOS 在 DOTA v1.0 上相对原始 YOLOv3 mAP 提高 7.04%，参数量约为原模型的 10.84%，模型大小约为 5.08%。
2. 在 VisDrone2019 上，Ghost-YOLOS 相对原始 YOLOv3 mAP 提高 4.54%，参数量约为原模型的 9.45%，模型大小约为 6.21%。
3. 在 Xilinx ZYNQ Z7020 上，CNN accelerator 达到 29.53 GOPS，功耗 2.98 W，DSP 使用 113 个，能效 9.90 GOPS/W，DSP 效率 0.261 GOPS/DSP。
4. 论文认为其方案适合低功耗、资源受限的遥感边缘设备。

#### 不足

论文自己给出了几个重要限制：

1. 剪枝和蒸馏方法强依赖具体模型，不能提升模型本身的检测上限。
2. 16-bit 量化仍会造成精度损失，需要更细粒度、更硬件友好的量化方法。
3. 受 Zynq-7020 资源限制，PE 并行度无法继续扩展，存在一定资源冗余。
4. 论文选择 YOLOv3 主要是硬件友好性考虑，而不是因为 YOLOv3 是精度最优网络。
5. 它把 backbone 作为轻量化重点，但对 Neck/FPN/PAN 中的跨层融合和 Concat 访存压力讨论不够深入。

### 2.4 与我的论文的联系

这篇论文和你的项目在平台和主题上很接近。

相同点：

```text
边缘检测任务
        ↓
YOLO 类网络
        ↓
轻量化 + FPGA 加速
        ↓
Zynq-7020 级别低功耗平台
        ↓
关注 GOPS/W、DSP efficiency、模型大小、mAP
```

差异点：

```text
Yang 2023：用 GhostNet 替换 backbone，再做通道剪枝、蒸馏、整网 accelerator。
你的论文：聚焦 YOLO/FPN 融合结构，用信息熵评价 FPN 分支贡献，再对 Concat + 3x3 Conv 关键融合算子做 HLS 流式重构。
```

可直接服务你的论文：

1. **Chapter 1**：支撑“遥感/工业/边缘检测模型难以部署在低功耗设备上”的背景。
2. **Chapter 2.1 Lightweight Object Detection**：GhostNet、通道剪枝、蒸馏、量化可以作为轻量化检测器代表。
3. **Chapter 2.3 FPGA-Based CNN Accelerators**：Zynq-7020、29.53 GOPS、2.98 W、9.90 GOPS/W 可进入 SOTA 表格候选。
4. **Chapter 3.3 Fixed-Point Quantization Strategy**：16-bit 定点量化是你选择 `ap_fixed<16,8>` 的文献依据之一，但你还需要做自己的位宽扫描。
5. **Research Gap**：它说明软硬件协同有效，但仍偏 backbone 轻量化，缺少对 FPN/Neck 跨层特征搬运的细粒度分析。

你的论文可以这样形成差异化表达：

```text
Existing lightweight detectors mainly reduce backbone computation or channel redundancy.
However, for YOLO-style detectors deployed on resource-limited FPGAs, the FPN/PAN neck also introduces non-negligible feature-map movement through skip connections and concatenation.
This work therefore focuses on entropy-guided FPN branch pruning and streaming reconstruction of the feature fusion operator.
```

## 3. Zhang et al., 2015 - Optimizing FPGA-based Accelerator Design for Deep Convolutional Neural Networks

### 3.1 论文解决的问题

这篇论文解决的是：FPGA CNN 加速器设计空间很大，计算吞吐和片外带宽经常不匹配，导致资源利用率低或性能达不到最优。

论文指出，一个 CNN 卷积层在 FPGA 上可以有很多实现方式：

```text
不同 loop tiling
不同 loop unrolling
不同 loop scheduling
不同 buffer 大小
不同片外带宽需求
不同 PE 并行度
```

即使资源利用率相同，不同设计的性能也可能差很多。核心矛盾是：

```text
如果 PE 太多但 DDR 喂不饱
        ↓
计算资源空转，memory-bound

如果片外带宽足够但 PE 太少
        ↓
带宽没用满，compute-bound
```

因此论文用 Roofline Model 来分析“计算能力”和“访存能力”的匹配关系。

### 3.2 核心解决方法

#### 3.2.1 总体路线图

```text
输入：CNN 卷积层参数 N, M, R, C, K, S 和 FPGA 平台资源/带宽
        ↓
对卷积循环做 tiling，定义 Tm, Tn, Tr, Tc
        ↓
对可并行维度做 loop unrolling 和 loop pipelining
        ↓
用 local memory promotion 减少片外访问
        ↓
计算每种设计的 CTC ratio 和所需带宽
        ↓
用 Roofline Model 判断 compute-bound / memory-bound
        ↓
筛选平台支持的设计点
        ↓
对各层做最优选择，并枚举 cross-layer uniform unroll factor
        ↓
生成统一 CNN accelerator
```

#### 3.2.2 算法一：Roofline Model

Roofline Model 的核心公式是：

```text
Attainable Performance = min(Computational Roof, CTC Ratio x Bandwidth)
```

其中：

```text
Computational Roof：平台可提供的最高计算吞吐
Bandwidth：片外存储带宽
CTC Ratio：Computation to Communication Ratio，每搬运一个单位数据能做多少计算
```

解释：

```text
如果 CTC Ratio 很低
        ↓
每做一点计算就要大量访问 DDR
        ↓
性能受带宽限制，即 memory-bound

如果 CTC Ratio 足够高
        ↓
DDR 访问被摊薄
        ↓
性能主要受 PE 数量和频率限制，即 compute-bound
```

这对你的论文非常重要，因为 Line Buffer / Window Buffer 的本质就是提高数据复用，从而提高 CTC Ratio。

#### 3.2.3 算法二：Loop Tiling

卷积层原始循环通常包含：

```text
输出通道 M
输入通道 N
输出行 R
输出列 C
卷积核行 K
卷积核列 K
```

由于整层 feature map 和权重通常无法全部放入片上，论文引入 tile size：

```text
Tm：一次处理多少个输出通道
Tn：一次处理多少个输入通道
Tr：一次处理多少行输出
Tc：一次处理多少列输出
```

Loop Tiling 的目标：

```text
把大卷积拆成可以放入片上 buffer 的小块
        ↓
在片上对小块进行充分复用
        ↓
减少 DDR 反复读取
```

但 tile size 不是越大越好。太大可能超过 BRAM；太小又会降低复用，增加 DDR 访问。因此 tile size 是设计空间搜索变量。

#### 3.2.4 算法三：Loop Unrolling 和 Loop Pipelining

Loop Unrolling：

```text
把某些循环维度展开成并行硬件
        ↓
例如展开 Tm 和 Tn
        ↓
多个输出通道 / 输入通道同时计算
```

Loop Pipelining：

```text
让循环不同迭代重叠执行
        ↓
目标是缩短 initiation interval, II
        ↓
理想情况下每个周期启动一次新计算
```

关键问题是依赖关系。若循环存在 loop-carried dependence，就无法充分 pipeline。论文用 polyhedral-based analysis 判断哪些循环可以合法重排，从而把更适合并行的循环放到内层。

#### 3.2.5 算法四：Local Memory Promotion 和数据复用

**解决什么问题：** 如果每次卷积窗口移动都从 DDR 重新读输入和权重，带宽会爆炸。

Local Memory Promotion 的思想：

```text
发现某些数组在内层循环中可以复用
        ↓
把这些数组从 DDR 提升到片上 buffer
        ↓
多个循环迭代共享同一批片上数据
        ↓
减少 DDR 访问次数
```

对卷积而言，可复用的数据包括：

```text
input feature tile
weight tile
partial output tile
```

这与本项目中的 Line Buffer / Window Buffer 思想一致：把相邻窗口重复需要的像素留在片上，而不是每个 3x3 窗口都重新读 DDR。

#### 3.2.6 算法五：Design Space Exploration 和 Cross-layer Optimization

每种 tile size 和 loop order 都会对应一个设计点：

```text
设计点 P = <Tm, Tn, Tr, Tc, loop schedule>
```

对每个设计点计算：

```text
所需 BRAM
所需 DSP / PE
所需片外带宽
CTC Ratio
可达到吞吐
```

然后用 Roofline 筛选：

```text
1. 去掉 BRAM / DSP 超限的点
2. 去掉所需带宽超过平台带宽的点
3. 在剩余点中找吞吐高且资源合理的点
```

论文还讨论 cross-layer design。若每层都使用不同的最优 unroll factor，硬件会很复杂；因此它枚举一个跨层统一的 `<Tm, Tn>`，使同一个 accelerator 可以处理多个卷积层。

#### 3.2.7 算法六：Ping-pong Buffer 和外部数据传输引擎

硬件实现中使用双缓存：

```text
Buffer A 供 PE 计算当前 tile
Buffer B 同时从 DDR 加载下一 tile
        ↓
下一阶段交换 A/B 角色
```

这样可以把数据传输时间和计算时间重叠起来。论文还用多个 AXI4 接口提升有效 DDR 带宽，并实测单纯增加接口位宽不一定提升带宽，增加 IP-AXI 接口数量更有效。

### 3.3 论文结论与不足

#### 结论

1. CNN FPGA 加速不能只堆 PE 或只减少 DDR 访问，必须匹配计算吞吐和片外带宽。
2. Roofline Model 可以帮助判断设计是 memory-bound 还是 compute-bound。
3. Loop tiling、loop unrolling、loop pipelining 和 local memory promotion 可以系统性地构建设计空间。
4. 论文在 Xilinx VC707 上实现 CNN accelerator，100 MHz 下达到 61.62 GFLOPS；相对 CPU 单线程有 17.42x 加速，相对 16 线程 CPU 也有约 4.8x 加速。
5. FPGA 运行功耗约 18.61 W，能耗相比 CPU 实现明显更低。

#### 不足

1. 论文主要面向早期 CNN/AlexNet 风格卷积层，没有覆盖 YOLO/FPN/PAN 的复杂分支、Concat 和多尺度融合。
2. 它主要优化卷积层，池化、全连接等其他层在文中被放到 future work 或外围处理。
3. 论文使用 VC707/Virtex-7 级别平台，与本项目 Zynq-7020 的资源规模不同，结果不能直接对比。
4. 论文重点是浮点或通用 CNN 卷积加速器设计，现代低比特量化、检测头、Neck 结构和端到端目标检测指标不是重点。
5. Cross-layer uniform accelerator 降低了硬件复杂度，但也可能牺牲某些层的局部最优性能。
